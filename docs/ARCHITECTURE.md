## 2. Naming Conventions

### Files and Folders

| Type | Convention | Example |
|------|-----------|---------|
| Page files | `page.tsx` (Next.js convention) | `app/dashboard/page.tsx` |
| Layout files | `layout.tsx` | `app/(auth)/layout.tsx` |
| API routes | `route.ts` | `app/api/weather/daily/route.ts` |
| React components | PascalCase `.tsx` | `RevenueEntryForm.tsx` |
| Utility/lib files | camelCase `.ts` | `format.ts`, `supabase-server.ts` |
| Hooks | `use` prefix, camelCase | `useWeather.ts` |
| Folders | kebab-case | `daily-summary/`, `invite-user/` |

### React Components

- One component per file
- Export as named export AND default export from component files
- Props interface defined inline above the component, named `[ComponentName]Props`

```tsx
// Good
interface RevenueEntryFormProps {
  siteId: string
  date: string
  existingData?: DailyRevenue
}

export function RevenueEntryForm({ siteId, date, existingData }: RevenueEntryFormProps) {
  // ...
}

export default RevenueEntryForm
```

### Variables and Functions

- `camelCase` for variables and functions
- `PascalCase` for types, interfaces, and enums
- `SCREAMING_SNAKE_CASE` for true constants (rarely needed)
- Descriptive names — no abbreviations except well-known ones (`id`, `url`, `ZAR`)

## 3. TypeScript Patterns

### tsconfig.json

```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true
  }
}
```

### Core Rules

- **No `any`** — use `unknown` and narrow with type guards or Zod
- **No redundant type annotations** — let TypeScript infer where it can
- **Use `satisfies`** for config objects to catch errors without widening the type
- **Discriminated unions** for state machines and variant types
- **Zod at all API boundaries** — validate all request bodies

### Discriminated Unions

```ts
// User roles
type UserRole = 'admin' | 'manager'

// Cost categories
type CostCategory = 'cos' | 'capex'

// Staff roles
type StaffRole = 'owner' | 'supervisor' | 'washer'

// Loading state pattern
type AsyncState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; error: string }
```

### `satisfies` for Config

```ts
const defaultServiceTypes = [
  { name: 'Basic Wash', active: true },
  { name: 'Full Wash', active: true },
  { name: 'Valet', active: true },
] satisfies { name: string; active: boolean }[]
```

### Zod at API Boundaries

```ts
import { z } from 'zod'

const LogCostSchema = z.object({
  date: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
  amount: z.number().positive(),
  category: z.enum(['cos', 'capex']),
  description: z.string().min(1),
  notes: z.string().optional(),
})

type LogCostInput = z.infer<typeof LogCostSchema>
```

### Type Guards

```ts
function isPostgrestError(error: unknown): error is { message: string; code: string } {
  return (
    typeof error === 'object' &&
    error !== null &&
    'message' in error &&
    'code' in error
  )
}
```

## 5. Supabase Client Rules

### Never mix client types

| Context | Client to use | Import from |
|---------|--------------|-------------|
| Browser (client components) | `createBrowserClient` | `src/lib/supabase.ts` |
| Server (page, layout, API route, server action) | `createServerClient` | `src/lib/supabase-server.ts` |
| Admin operations (invite/remove user) | `createClient` with service key | Inside API route only — never expose |

### `src/lib/supabase.ts` (browser)

```ts
import { createBrowserClient } from '@supabase/ssr'
import type { Database } from './database.types'

export function createClient() {
  return createBrowserClient<Database>(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
  )
}
```

### `src/lib/supabase-server.ts` (server)

```ts
import { createServerClient } from '@supabase/ssr'
import { cookies } from 'next/headers'
import type { Database } from './database.types'

export async function createClient() {
  const cookieStore = await cookies()
  return createServerClient<Database>(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() { return cookieStore.getAll() },
        setAll(cookiesToSet) {
          try {
            cookiesToSet.forEach(({ name, value, options }) =>
              cookieStore.set(name, value, options)
            )
          } catch {}
        },
      },
    }
  )
}
```

## 6. API Route Patterns

All API routes follow this structure:

```ts
import { NextRequest, NextResponse } from 'next/server'
import { createClient } from '@/lib/supabase-server'
import { z } from 'zod'

// Zod schema for request body (POST routes)
const RequestSchema = z.object({
  // ...
})

export async function POST(request: NextRequest) {
  // 1. Auth check
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  }

  // 2. Parse and validate body
  const body: unknown = await request.json()
  const result = RequestSchema.safeParse(body)
  if (!result.success) {
    return NextResponse.json({ error: 'Invalid request' }, { status: 400 })
  }

  // 3. Business logic
  // ...

  // 4. Return response
  return NextResponse.json({ success: true })
}
```

## 7. Server Actions vs API Routes

| Use API Route when... | Use Server Action when... |
|----------------------|--------------------------|
| External services call it (webhooks) | Form submissions from client components |
| You need fine-grained HTTP status codes | Simple CRUD triggered by user interaction |
| Admin operations (user invite/remove) | Settings saves, attendance marking |
| Weather fetch and cache | Revenue entry, cost logging |

Server actions live in the same file as the page or a co-located `actions.ts` file.

## 9. Chart Component Pattern

All Recharts components **must** follow this pattern to avoid SSR errors:

```tsx
// src/components/dashboard/RevenueBarChart.tsx
'use client'

import dynamic from 'next/dynamic'
import { BarChart, Bar, XAxis, YAxis, Tooltip, ResponsiveContainer } from 'recharts'
import { formatZAR } from '@/lib/format'

// The component itself has 'use client' and is used inside a dynamic() wrapper
// when imported into page.tsx

interface RevenueBarChartProps {
  data: { date: string; total: number }[]
}

export function RevenueBarChart({ data }: RevenueBarChartProps) {
  return (
    <ResponsiveContainer width="100%" height={240}>
      <BarChart data={data}>
        <XAxis dataKey="date" />
        <YAxis tickFormatter={(v: number) => formatZAR(v)} />
        <Tooltip formatter={(v: number) => formatZAR(v)} />
        <Bar dataKey="total" fill="var(--color-sky-500)" />
      </BarChart>
    </ResponsiveContainer>
  )
}
```

In the page that uses it:

```tsx
// src/app/dashboard/page.tsx
import dynamic from 'next/dynamic'

const RevenueBarChart = dynamic(
  () => import('@/components/dashboard/RevenueBarChart').then(m => m.RevenueBarChart),
  { ssr: false, loading: () => <ChartSkeleton /> }
)
```

## 11. Error Handling

### In API routes — never expose internals

```ts
try {
  // ...
} catch (error) {
  console.error('Revenue summary error:', error) // log internally
  return NextResponse.json(
    { error: 'Failed to fetch revenue summary' }, // generic to client
    { status: 500 }
  )
}
```

### In client components — show inline errors

```tsx
const [error, setError] = useState<string | null>(null)

// On failure:
setError('Failed to save. Please try again.')

// In JSX:
{error && (
  <div role="alert" className="rounded-lg bg-red-50 p-3 text-sm text-red-700">
    {error}
  </div>
)}
```

## 12. `site_id` Rule

Every database insert must include `site_id`. Every query is automatically filtered by RLS, but always include it explicitly on insert to be safe and clear.

```ts
// Good
await supabase.from('costs').insert({
  site_id: profile.site_id,   // always explicit
  date,
  amount,
  category,
  description,
  created_by: user.id,
})
```

## 13. Import Aliases

Configure `tsconfig.json` with:

```json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

Use `@/` imports everywhere. Never use relative `../../` paths except within the same folder.

```ts
// Good
import { formatZAR } from '@/lib/format'
import { Button } from '@/components/ui/Button'

// Bad
import { formatZAR } from '../../../lib/format'
```
